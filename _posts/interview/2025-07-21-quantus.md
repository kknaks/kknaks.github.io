## 💼 도메인
### 퀀트 투자 
- **정의** : Quantitative Investment로 정량적인 수치와 데이터를 통계적으로 분석하고 자동화된 투자하는 전략

  - 백테스트 : 과거데이터로 전략의 성과를 검증

## 💻 과제질문
### 🏗️ 아키텍처 & 설계

#### 기본 설계 질문
> **FastAPI를 선택한 이유는 무엇인가요? Django나 Flask 대신 FastAPI를 사용한 특별한 이유가 있나요?**
> 개발 생산성 : 7일이란 과제 시간에 ORM설정, Admin 생성과 같은 설정이 복잡
> 타입 안정성 : 처음으로 pandas와 fastAPI를 결합을 시도 했는데 타입에러를 줄이기 위해 선택
> 비동기 처리 : 현재 기업별 재무제표를 ThreadPoolExecutor로 병렬 처리, 향후 비동기 처리 강화(현재 멀티스레딩 방식으로 OS레벨에서 스레드를 생성, 단일 쓰레드에서 이벤트 루프 기반으로 실행 하는 방식)
> 트레이드 오프 : async 미지원 라이브러리, 동기/비동기 혼용으로 복잡도 증가

> **현재 구현된 레이어드 아키텍처(Controller-Service-API)의 장단점을 설명해주세요.**
> 관심사 분리 : Controller는 HTTP 요청/응답, 비즈니스로직은 Service, API는 외부 통신용
> 재사용성 : 여러조합을 재사용 가능 
> 성능 오버헤드 : DataFrame → Pydantic → DataFrame 구조로 메모리 사용량이 많음 
> 순환 의존성 위험 : 서비스들이 서로 참조하면서 의존성 문제 발생 가능성 -> DI 컨테이너 도입

> **서비스가 커질 때 이 구조를 어떻게 확장하시겠나요? 마이크로서비스로 분리한다면 어떤 기준으로 나누겠나요?**
> DDD원리에 따라 도메인 별로 구분 (데이터 수집, 데이터 분석, 백테스트)
> 데이터 수집 : 외부 데이터 수집로직, 데이터 전처리
> 데이터 분석 : 재무비율 계산, 이상치 제거, 통계 분석, CPU 집약적으로 별도 스케쥴링이 필요
> 백테스트 : 수익률 분석, 장시간 실행되는 비동기 작업 특화 

#### 심화 설계 질문
> **현재 코드에서 `KrxApi`와 `DartApi`가 싱글톤 패턴으로 구현되어 있는데, 이 선택의 이유와 잠재적 문제점은 무엇인가요?**
> 지금 싱글톤으로 만드려고 했지만 미완성 단계, 싱글톤 패턴의 문제는 여러 쓰레드가 객체를 동시에 생성 할 수 있음. 
> 선택의 이유 : API키 관리 효율성과 HTTP 연결 풀을 재사용
> DI 컨테이너 도입을 필요 

> **백테스트 서비스에서 대량의 금융 데이터를 처리할 때 메모리 최적화는 어떻게 하시겠나요?**
> 현재 DB를 사용 하지 않고 있어서 메모리 누수가 발생
> PostgreSQL과 같은 DB를 사용해서 데이터 조회(금융 데이터 중 필요한 컬럼만 조회)
> pandas를 사용하던 부분을 Pydantic모델을 사용해서 메모리 효율성 증가 

> **투자 지표 계산 로직이 복잡한데, 이를 더 모듈화하고 테스트 가능하게 만들려면 어떻게 리팩토링하시겠나요?**
> 지표별 클래스 분리
> 팩토리 패턴 적용
> 의존성 주입

### 🚀 성능 & 최적화

#### 기본 성능 질문
> **현재 코드에서 ThreadPoolExecutor를 사용하는 부분이 있는데, 언제 멀티스레딩을 사용하고 언제 비동기를 사용해야 하나요?**
> 멀티쓰레딩 : 여러쓰레드가 동시작업, 메모리공유, CPU집약적 작업에 적합
> 비동기 : 단일 쓰레드에서 이벤트루프로 논블로킹 처리, I/O 바운드 작업에 적합
> 현재 코드는 threadPool을 전부 사용하고 있지만, 기업주가를 불러오는 api 비동기, 각종 지표를 생성하고 분석하는 경우는 멀티쓰레딩이 적합

> **DART API 호출 시 Rate Limiting을 어떻게 처리하고 있나요? 더 효율적인 방법은 없을까요?**
> 현재 상황 : 멀티쓰레딩을 워커를 제한하여 동시요청을 보내고 있음 -> API 제공 업체측의 제한 사항을 고려하지 않음
> 개선 방안 : 429응답에 대한 지수백오프 (오류 시 요청 인터벌 증가), 일일 호출제한량 카운팅

> **Pandas DataFrame 연산이 많은데, 대용량 데이터에서 성능 이슈가 발생한다면 어떻게 해결하시겠나요?**
> 현재 상황 : 데이터 분석을 위해 pandas를 사용 한것을 그대로 차용 
> 개선 방안 : DB모델링을 통해 인덱싱을 하고 Pydantic을 활용하여 데이터를 구조화 및 타입안정성을 높으는 전략으로 수정

#### 심화 성능 질문
> **백테스트에서 월별 리밸런싱을 수행할 때 O(n²) 복잡도가 발생할 수 있는데, 이를 최적화하는 방법은?**
> 개선방안1 : pandas를 사용한다면 행 전체를 필터링해서 결과를 가져와서 데이터 규모를 줄인다. O(n^2) 복잡도 소요
> 개선방안2 : DB모델링을 통해 스크리닝 결과를 한번의 쿼리로 가져온 후 연산을 수행 O(nlogn) 복잡도 소요

> **캐싱 전략을 구현한다면 어떤 데이터를 캐싱하고, 어떤 저장소(Redis, Memcached)를 선택하시겠나요?**
> Dart, KRX API : 비교적 호출 빈도가 적은 데이터, 과거 데이터로 변경 빈도가 낮음.
> 적용 방안 : 최초 API 호출 시 DB에 데이터 저장, 향후 재 사용시 DB에서 redis로 캐싱 후 사용 
> Redis : TTL 설정과 복잡한 데이터 구조 저장이 용이하기 때문
> 캐시 무효화 처리 : 분기별 데이터는 TTL을 3개월로 설정하고, 신규 분기 발표 시 관련 캐시를 수동으로 무효화
> 메모리 사용량이 증가 대책 : LRU 정책으로 오래된 데이터를 자동 삭제하고, 자주 조회되는 데이터만 Redis에 유지

> **데이터베이스를 도입한다면 현재 CSV 파일 기반 저장을 어떻게 마이그레이션하시겠나요?**
> 모델 객체를 만들고 alembic을 통해 데이터 주입
> 트랜잭션 처리 : csv데이터 검증 후, 배치 단위로 트랜잭션 커밋 및 롤백을 진행

### 🛡️ 에러 처리 & 안정성

#### 기본 안정성
> **외부 API 호출 실패 시 재시도 로직이 구현되어 있는데, 이 방식의 개선점은 무엇인가요?**
> API 호출 제한량 실패 원인은 일일 호출 제한, 분당 호출 제한이 있음
> 분당 호출 제한을 피하기 위해서는 재시도 로직에서 인터벌을 추가하여 다시 호출을 시도

> **`HTTPException`을 일관성 있게 사용하고 있는데, 커스텀 Exception 클래스를 만든다면 어떻게 설계하시겠나요?**
> 기본 분류 : 각 도메인별 에러를 분류
> 기본 정보 : 에러 코드, 메시지, 발생 위치 표시 
> 위험도 분류 : critical, warning, recoverable로 분류 


#### 심화 안정성
> **Circuit Breaker 패턴을 적용한다면 어떤 부분에 적용하시겠나요?**
> circuit breaker 패턴 : open, closed, half-open 상태를 활용하여 연쇄적인 시스템 장애 방지
> 적용 방안 : API 호출 시, 과도한 트래픽이 걸렸을 경우 적용 -> 요청이 30%이상 실패시 open으로 전환하고 1분 이후 half-open으로 전환하여 상태를 확인 후 closed로로 최종 전환

> **데이터 일관성을 보장하기 위한 트랜잭션 처리는 어떻게 구현하시겠나요?**
> 트랜잭션이 필요한 상황 : 재무데이터+주가 데이터 배치 저장(대용량), 백테스트 결과
> 트랜잭션 범위 : 단일테이블(재무데이터, 주가), 다중테이블(기업정보, 각종 지표), 배치 트랜젝션(대량 데이터 처리)
> 격리 수준 : 데이터의 일관성이 중요하므로 Serializable로 사용

> **시스템 장애 시 백테스트 작업을 이어서 수행할 수 있는 방법은?**
> 해야할 작업을 큐에 저장 작업이 완료시 큐에서 작업
> 체크포인트를 만들어서 백테스트 진행 상황을 기록
> 캐시DB를 활용해서 분기별 진행 상황을 캐싱
> 미 완료시 재시도 로직 구현
